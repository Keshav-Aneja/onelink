Feature Plan Effort: Medium-High

RSS Feeds Dashboard

Making RSS a first-class citizen in OneLink — a dedicated Feeds section with subscription management, unread tracking, OPML import/export, and feed health monitoring. Zero extra cost, zero AI/LLM dependencies.

Problem Statement

⚠️
Hidden subscription model. Users subscribe per-link with no central view. There is no way to see all subscribed feeds at once, or know which feeds are healthy vs. broken.
⚠️
No read/unread state. Every notification-list visit re-fetches items from the past N days. Items already seen look identical to new ones — no sense of progress.
⚠️
No portability. There is no way to import subscriptions from Feedly/NewsBlur (OPML), nor to back up and restore your feed list.
⚠️
Flat, unsorted feed. All updates from all sources land in one undifferentiated list. No per-source grouping, no "Mark all read" shortcut per feed.

Proposed Solution

✅
A dedicated "Feeds" route that surfaces all subscribed RSS sources as manageable cards — with last-fetched time, unread count, and health status — plus OPML import/export and per-feed "Mark all read".

Unread state is tracked server-side in a new lightweight rss_read_items table (just a set of seen item hashes per user). No AI, no third-party services, no added infrastructure cost.
📋

Feed Manager

All subscribed feeds as cards. Name, domain favicon, unread count badge, last-fetched time, health indicator, and quick-unsubscribe.

🔴

Unread Tracking

Each fetched item gets a stable hash. A lightweight rss_read_items table marks seen items per user. No external service needed.

📤

OPML Import / Export

Parse standard .opml XML client-side, batch-subscribe backend. Export generates an opml blob server-side — fully portable.

❤️‍🩹

Feed Health

Track last_fetched_at and last_error on each subscription row. Surface a warning icon when a feed hasn't updated in 7+ days.

✔️

Mark All Read

Per-feed and global "mark all read" action — bulk-inserts item hashes into rss_read_items. Instant UX with optimistic updates.

🔗

Decoupled Subscriptions

New rss_subscriptions table fully decoupled from the links table. A feed can be subscribed without a saved link (pure RSS reader mode).

Database Schema Changes

New table: rss_subscriptions

CREATE TABLE rss_subscriptions ( id UUID PRIMARY KEY, owner_id UUID NOT NULL REFERENCES users(id), feed_url TEXT NOT NULL, site_url TEXT, -- canonical page URL title TEXT, favicon_url TEXT, link_id UUID REFERENCES links(id) ON DELETE SET NULL, last_fetched_at TIMESTAMPTZ, last_error TEXT, -- null = healthy created_at TIMESTAMPTZ DEFAULT NOW(), UNIQUE(owner_id, feed_url) ); CREATE INDEX ON rss_subscriptions(owner_id);

New table: rss_read_items

CREATE TABLE rss_read_items ( owner_id UUID NOT NULL REFERENCES users(id), item_hash TEXT NOT NULL, -- SHA-1( feed_url || item_guid || pub_date ) read_at TIMESTAMPTZ DEFAULT NOW(), PRIMARY KEY(owner_id, item_hash) ); CREATE INDEX ON rss_read_items(owner_id); -- Automatic cleanup: items older than 90 days -- are irrelevant. Run via a weekly cron or -- a Postgres rule (no extra service needed).

Migration note on links table

-- links.subscribed + links.rss are kept for -- backwards compatibility but deprecated. -- New subscriptions go into rss_subscriptions. -- A back-fill migration creates rss_subscriptions -- rows for every link where subscribed=true. ALTER TABLE links ADD COLUMN rss_subscription_id UUID REFERENCES rss_subscriptions(id) ON DELETE SET NULL;
ℹ️
Why a separate table? The current model ties RSS subscriptions to saved links 1-to-1. A user might want to subscribe to a feed without saving a link (pure reader mode), or a single feed URL might appear in multiple link entries. Decoupling removes both limitations and avoids duplicating fetch work across link copies.

Backend — New & Modified Endpoints

Method Route Description Status
GET /feeds Return all rss_subscriptions for the authenticated user, joined with latest last_fetched_at and unread count (COUNT of items not in rss_read_items — computed from last cached fetch). New
POST /feeds Subscribe to a feed URL. Auto-discovers the RSS/Atom URL if a page URL is provided (reuses existing RSS.findValidRSS() logic). Returns the new rss_subscriptions row. New
DELETE /feeds/:id Unsubscribe from a feed. Removes rss_subscriptions row; sets links.subscribed=false and links.rss_subscription_id=null for any linked link rows. New
POST /feeds/:id/read Mark all current items in a feed as read. Body: { item_hashes: string[] }. Bulk-inserts into rss_read_items with ON CONFLICT DO NOTHING. New
POST /feeds/read-all Global "mark all read" — marks every currently-fetched item across all user feeds as read. New
GET /feeds/opml Export all subscriptions as a valid OPML 2.0 XML document. Generated server-side from rss_subscriptions. No external dependency. New
POST /feeds/opml Import an OPML file. Parse XML, extract <outline xmlUrl> entries, deduplicate against existing subscriptions, bulk-create rss_subscriptions rows. Returns count added. New
POST /links/feed Existing feed aggregation endpoint. Extend to also accept feedId for per-feed filtering, and return item_hash alongside each item so the frontend can track read state. Modified
PATCH /links/:id Existing update endpoint. When subscribed=true, create an rss_subscriptions row (if not exists) and set links.rss_subscription_id. Modified

Unread Tracking — Item Hash Strategy

🔑
Hash inputs (in order of preference):
  • RSS <guid isPermaLink="false">
  • Atom <id> element
  • Fallback: SHA-1(feed_url + item.link + item.pub_date)

Hashes are computed in the scraper (packages/scraper/module/rss/index.ts) and returned alongside the item. The backend stores them only in rss_read_items — never the full article content.
💾
Storage estimate: A user with 20 feeds, each publishing 10 items/week, generates ~800 hash rows/month. At ~60 bytes per row, that is under 50 KB/month per user. A 90-day TTL cleanup keeps the table bounded. No external storage, no object storage, no message queue.

Data Flow — Key Sequences

1. Opening the Feeds page

Step 1

GET /feeds

Frontend calls new endpoint. Returns subscription rows + unread count (derived from cached item list vs. read hashes).

→
Step 2

Render feed cards

Each subscription card shows title, favicon, unread badge, last_fetched_at, and health icon.

→
Step 3

Click feed card

GET /links/feed?feedId=<id> returns items for that feed with item_hash. Items not in read set are marked bold/unread.

→
Step 4

Mark as read

On item open or "Mark all read", POST /feeds/:id/read with item_hashes. Optimistic update in React Query.

2. OPML import flow

Step 1

Client parse

Browser parses XML with DOMParser. Extracts xmlUrl from each <outline>. Zero server round-trip for preview.

→
Step 2

Preview modal

Show list of feeds found. Check against current subscriptions. User confirms.

→
Step 3

POST /feeds/opml

Send raw OPML XML. Backend parses, deduplicates, bulk-inserts into rss_subscriptions.

→
Step 4

Invalidate cache

React Query invalidates ["feeds"] key. Feeds page refreshes with new subscriptions visible immediately.

3. Feed health tracking (server-side, no extra service)

Step 1

Existing scrape call

When POST /links/feed runs, the scraper already calls each RSS URL. No new scheduling needed.

→
Step 2

Update subscription row

On success: set last_fetched_at=now(), last_error=null. On failure: set last_error=message.

→
Step 3

Surface in UI

GET /feeds returns last_fetched_at and last_error. Frontend shows green/red health icon accordingly.

Zero-Cost & Zero-AI Design Decisions

🔒

No new services

All RSS fetching uses the existing packages/scraper module already in the codebase. The Redis cache layer is already in place. No message queue, no background worker daemon.

🧮

No new libraries (frontend)

OPML parsing uses the browser's built-in DOMParser. SHA-1 hashing uses Node's built-in crypto module. No npm packages to add, no bundle size increase.

🗄️

Minimal DB footprint

rss_read_items stores only short hashes (~40 chars) + timestamps. 90-day TTL keeps rows bounded. Even at scale this stays in kilobytes per user per month.

🤖

No AI/LLM anywhere

Feed discovery already works via the scraper's deterministic path-checking + site-specific patterns. Unread detection is a set-difference query. No model inference at any step.
